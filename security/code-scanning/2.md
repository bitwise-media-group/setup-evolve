# Code scanning finding #2 — Network data written to file

**Finding:**
[github.com/bitwise-media-group/setup-evolve/security/code-scanning/2](https://github.com/bitwise-media-group/setup-evolve/security/code-scanning/2)

| Field    | Value                                          |
| -------- | ---------------------------------------------- |
| Rule     | `js/http-to-file-access` (CWE-434, CWE-912)    |
| Tool     | CodeQL 2.25.6                                  |
| Severity | Warning (security-severity: medium)            |
| Location | `__tests__/integration/real-verify.test.ts:43` |
| Detected | 2026-06-16 against commit `2cba6cd`            |
| Outcome  | **False positive**                             |

## What CodeQL reported

The query traces a `fetch` response body to a file-system write and warns that storing network data on disk "allows
arbitrary file upload and might indicate a backdoor". The source is the `fetch` at line 39 and the sink is the
`writeFileSync` at line 43:

```javascript
const url = `https://github.com/bitwise-media-group/evolve/releases/download/v${VERSION}/${target.asset}`;
const res = await fetch(url);
expect(res.ok).toBe(true);
archive = join(dir, target.asset);
const { writeFileSync } = await import("node:fs");
writeFileSync(archive, Buffer.from(await res.arrayBuffer()));
```

## Why this is a false positive

**1. It is test-only code, classified as such by CodeQL itself.**

The alert carries the `test` classification. The file is a gated integration test
(`describe.skipIf(!process.env.RUN_INTEGRATION …)`) that runs only when `RUN_INTEGRATION=1` is set explicitly — never in
the shipped Action (`dist/`) and never in the default test run. It is not part of any runtime code path a consumer of
this Action executes.

**2. The URL is a fixed, trusted constant — not attacker-controlled input.**

The backdoor pattern this query targets is _untrusted, attacker-influenced_ data written to a path other code later
trusts. Here the URL is a hard-coded literal pointing at the project's own signed release
(`github.com/bitwise-media-group/evolve/releases/download/v0.1.0/…`) over HTTPS. Nothing about the path or the content
is user-controlled; `target.asset` is selected from a static `PLATFORMS` table keyed on
`process.platform`/`process.arch`.

**3. The download is written to an ephemeral temp dir and exists only to be verified.**

The bytes land in a freshly created `mkdtempSync` directory, are extracted, and the extracted binary is then
**cryptographically verified** against the committed cosign bundle under the pinned signer identity (lines 60–63). The
same test asserts that a single flipped byte **fails** verification (lines 66–68). So far from "other code assuming the
local file can be trusted", the entire purpose of the test is to prove the downloaded bytes are _not_ trusted until they
pass signature verification — the opposite of the CWE-434/912 threat model.

## Resolution

No code change is warranted. The construct is a deliberate, gated integration test that downloads a fixed, trusted
release asset into a throwaway temp directory for the sole purpose of exercising signature verification. The finding is
dismissed as a **false positive** (GitHub dismissal reason: "Used in tests").

The repo's CodeQL config (`.github/codeql/codeql-config.yaml`) already scopes analysis to security queries over
`src/`-derived code; if test-classified findings continue to add noise, a future option is a `paths-ignore` entry for
`__tests__` — weighed against losing security coverage of test helpers.
