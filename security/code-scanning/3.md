# Code scanning finding #3 — Potential file system race condition

**Finding:**
[github.com/bitwise-media-group/setup-evolve/security/code-scanning/3](https://github.com/bitwise-media-group/setup-evolve/security/code-scanning/3)

| Field    | Value                                              |
| -------- | -------------------------------------------------- |
| Rule     | `js/file-system-race` (CWE-367)                    |
| Tool     | CodeQL 2.25.6                                      |
| Severity | Warning (security-severity: high)                  |
| Location | `src/install.ts:64` (check at `src/install.ts:60`) |
| Detected | 2026-06-16 against commit `2cba6cd`                |
| Outcome  | **Remediated**                                     |

## What CodeQL reported

A time-of-check/time-of-use (TOCTOU) race: the code checked the binary's existence with `fs.access` and then operated on
the same path by name, so "the file may have changed since it was checked". The check was line 60 and the use was line
64:

```javascript
const extractDir = await extractArchive(archivePath, platform.os)
const binaryPath = path.join(extractDir, binaryName(platform.os))
await fs.access(binaryPath)                                   // check (line 60)

const bundlePath = await downloadReleaseAsset(sigBundleAsset, token)
const bundleJson: unknown = JSON.parse(await fs.readFile(bundlePath, 'utf8'))
const binary = await fs.readFile(binaryPath)                  // use   (line 64)
const cosign = await verifyCosignBundle(bundleJson, binary)
```

## Assessment

The flagged pattern is real but its exploitability here is low:

- **The path is a run-private temp dir, not a shared location.** `extractDir` is created by `@actions/tool-cache` under
  `RUNNER_TEMP` for this workflow run. The query's motivating example is a world-writable `/tmp` in a multi-user system;
  a GitHub-hosted runner is a single-tenant ephemeral VM with no concurrent attacker process.
- **The bytes are cryptographically verified after reading.** The binary read at the "use" is immediately checked
  against the cosign bundle under a pinned signer identity (`verifyCosignBundle`). A binary swapped in via a race would
  fail verification, so a successful race cannot yield an unverified binary on the install path.

Even so, this is our own source (not vendored, unlike [1.md](./1.md)), and the check was **redundant**: `fs.access` only
probed existence, and the subsequent `fs.readFile` already throws `ENOENT` if extraction did not produce the binary. The
cheapest correct fix is to remove the separate probe entirely.

## Resolution

**Remediated** in `src/install.ts` (and the rebuilt `dist/` bundle): the `fs.access` check is removed and the binary is
read directly where the probe used to be.

```javascript
const extractDir = await extractArchive(archivePath, platform.os)
const binaryPath = path.join(extractDir, binaryName(platform.os))
// Read the extracted binary directly rather than probing with fs.access
// first: a separate check-then-use is a TOCTOU race, and readFile already
// throws (ENOENT) if extraction did not produce the expected binary.
const binary = await fs.readFile(binaryPath)

const bundlePath = await downloadReleaseAsset(sigBundleAsset, token)
const bundleJson: unknown = JSON.parse(await fs.readFile(bundlePath, 'utf8'))
const cosign = await verifyCosignBundle(bundleJson, binary)
```

This eliminates the check-then-use pair (only a single `readFile` "use" of the path remains, followed later by `chmod` —
neither is a check), so the data flow CodeQL traced no longer exists. Behaviour is preserved: the read still fails fast
with a clear `ENOENT` (including the path) before the signature bundle is downloaded, and the verified bytes are the
bytes held in memory through `verifyCosignBundle`. `npm run typecheck`, `npm test`, and `npm run build` (dist
reproducibility) all pass. The alert should resolve on the next analysis of `main`.
